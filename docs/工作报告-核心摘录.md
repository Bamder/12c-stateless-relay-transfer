## 0. 系统总览

本系统是一套面向不可信存储节点的端到端加密文件传输方案：用户侧完成加密与校验，中继只保存无法解读的密文分片；接收方凭通道外传递的 **12 位访问凭证**即可定位并还原文件，服务器无需也不应持有长期内容密钥。

### 组件

| 组件       | 路径                 | 职责                                                  |
| ---------- | -------------------- | ----------------------------------------------------- |
| 客户端     | `client/`          | WebAssembly 密码学 + TypeScript 传输编排；Web 界面    |
| 注册中心   | `server/registry/` | 块标识预留、放置规划、读路径排序、心跳与覆盖鉴权      |
| 中继节点   | `server/relay/`    | 本地对象存储 + 索引；写成功后向注册中心登记并定期上报 |
| 运维控制台 | `server/console/`  | 运维界面（本摘录不展开）                              |

### 拓扑与协作

采用常见的「控制面 + 数据面」拆分：一个注册中心负责索引与调度，多个中继负责存放对象。客户端先向注册中心**预留写入位置**或**查询读取位置**，再对指定中继执行对象的写入（PUT）或读取（GET）。中继在本地写入成功后向注册中心登记；若同一标识上已有内容需覆盖，须先完成鉴权。

## 1. 协议（对象格式与密码学）

### 1.1 职责与边界

规定如何将明文文件变换为「密文分片 + 定长加密元数据 + 可派生的块标识」，以及 12 位凭证如何拆成**寻址半段**与**密钥半段**。HTTP 路由与集群调度不在协议范围内。实现以 C++ / OpenSSL 编写，经 WebAssembly 在浏览器中运行；服务端不持有密钥半段，因而无法解开元数据或内容密文。

### 1.2 规范要点

**载荷保护的两种模式**（摘自协议文档）：

> 采用定长加密元数据、AES-256-GCM 保护内容、基于分片的 Merkle 树做完整性校验，以及由凭证确定性导出的块标识。
>
> - **整包模式（V2）**：对整段填充后的明文做一次认证加密。
> - **分段模式（V2.1）**：按固定明文段长多次独立做认证加密；元数据线上长度与版本字段保持不变。

**编码数据流（发送）**

```text
文件明文
    -->（可选前缀零填充）
    --> 载荷加密（整包或分段）--> 密文流 C
    --> 按上限切分为数据分片 B_1 .. B_m
    --> 计算 Merkle 根 → 写入元数据
    --> 元数据定长序列化并加密
    --> 第 0 号对象 = 加密元数据 ‖ 末分片；其余对象各存一个分片
```

**解码数据流（接收）**

```text
凭证（通道外）+ 各对象字节（按派生标识拉取）
    --> 自第 0 号对象取出加密元数据（前 373 字节）并解密解析
    --> 读取分片数、块长、密文总长、段模式等字段
    --> 重组全部数据分片 B_1 .. B_m，校验 Merkle 根
    --> 解开文件加密密钥 → 按整包或分段模式解密得填充明文
    --> 去掉前缀填充 → 按原始长度输出文件字节
```

两端合起来即「凭证 + 全部对象字节 ⇔ 还原文件」；解码步骤严格依赖元数据中的段模式字段，不得仅凭文件大小猜测。

**凭证分工（通用说法）**

```text
前 6 位  → 用于派生块标识（寻址，与密钥无关）
后 6 位  → 经慢速密钥派生得到对称密钥材料，再包装文件加密密钥
```

    const Bytes& block,
    const std::size_t leaf_index,
    const Bytes& root_hash,

## 3. 注册中心（控制面）

### 3.1 职责

维护块标识与存放节点的索引，并在上传时做**放置规划**（条带化主副本宽度与副本深度）、在下载时按节点负载做**读路径优选**；同时在用户请求的保留期限与集群实际能力不一致时进行降级授予，并对对象覆盖签发消息认证码。摘录侧重调度逻辑，HTTP 路由仅作对照。

技术栈：Python · FastAPI · SQLite（异步）。

### 3.2 对外接口（一览）

| 方法 | 路径                            | 作用                           |

| POST | `/api/relay/reserve-tokens`   | 预留标识并返回放置结果         |
| POST | `/api/relay/resolve`          | 返回按负载排序的可读取节点列表 |
| POST | `/api/relay/register`         | 中继写入成功后登记             |

### 3.3 核心代码（调度）

#### （1）联合选取条带宽度与副本深度

先定参数 `(S, R)`：`S` 为条带宽度（主写并行度），`R` 为文件级副本深度。这一步只做规模选型，**还不划分节点池**。

```31:62:server/registry/registry_server/scheduling/placement.py
def choose_stripe_replica(
    *,
    block_count: int,
    healthy_count: int,
    stripe_target_relays: int,
    max_file_replica_count: int,
) -> StripeReplicaChoice:
    """Stripe–replica sizing: pick (S, R) that maximizes layout capacity.
    Score is S×(1+R). Among equal scores, prefer a larger S.
    R(S) is min(max_file_replica_count, (healthy_count − S) // S).
    """
    s_max = min(stripe_target_relays, block_count, healthy_count)
    if s_max < 1:
        return StripeReplicaChoice(stripe_count=1, replica_factor=0)

    best_s, best_r, best_score = 1, 0, 1
    for s in range(s_max, 0, -1):
        r = min(max_file_replica_count, max(0, (healthy_count - s) // s))
        score = s * (1 + r)
        if score > best_score or (score == best_score and s > best_s):
            best_s, best_r, best_score = s, r, score
    return StripeReplicaChoice(stripe_count=best_s, replica_factor=best_r)
```

说明：在健康节点数量约束下同时决定 `S` 与 `R`，目标取 `S×(1+R)`；同分时偏好更大的 `S`，以提高并行写入宽度。

## 4. 中继节点（数据面）

### 4.1 职责

按对象标识读写本地**不可解读的二进制对象**；不解析上层元数据或明文。对外提供 PUT/GET；成功写入后向注册中心登记；覆盖前必须完成鉴权与消息认证码校验；后台清理过期对象并定期上报占用率等指标。

### 4.2 消息流向

| 方向             | 接口                   | 说明             |
| ---------------- | ---------------------- | ---------------- |
| 客户端 → 中继   | `PUT /{token}`       | 写入对象         |
| 客户端 → 中继   | `GET /{token}`       | 读取对象         |
| 中继 → 注册中心 | 登记 / 覆盖鉴权 / 心跳 | 状态同步与写保护 |
